  ai_company에 에이전트 팀이 더 나을 부분

  딱 하나입니다 — MCP 접근 문제

  팀원들은 CLAUDE.md, MCP servers, skills를 자동으로 로드합니다. 즉 팀원도 Rube
  MCP를 직접 쓸 수 있어서, 오늘 발생한 "Tool Agent가 Rube에 접근 못해서
  not_connected로 잘못 표시" 문제가 자연스럽게 사라집니다.

  ---
  그럼에도 현재 방식이 더 나은 이유

  1. ai_company의 Expert들은 서로 대화할 필요가 없습니다

  에이전트 팀의 핵심 가치는 "팀원들이 서로 논의하며 발견사항을 공유"하는 것입니다.
   문서 예시가 "버그 가설 5명이 서로 반박", "PR 리뷰어들이 의견 공유" 같은
  케이스인 이유가 여기에 있습니다.

  ai_company의 Expert들은 파일 기반으로 공유하고 독립적으로 실행하면 충분합니다.
  expert-001이 생성한 JSON을 expert-002가 읽으면 되고, 실시간 논의가 필요한 구조가
   아닙니다.

  2. CEO 개입 구조와 맞지 않습니다

  에이전트 팀은 "리더가 위임하고 팀이 자율 조율"하는 방식입니다. ai_company의 핵심
   가치인 **Task Briefing (CEO가 태스크별로 Tool/레퍼런스를 검토·승인)**이 팀
  구조에서는 자연스럽게 끼어들기 어렵습니다. 팀원들이 자체 조율하다 보면 CEO 개입
  시점 통제가 복잡해집니다.

  3. 실험적 기능의 제한들이 많습니다

  - 세션 재개 불가 → ai_company는 세션 간 상태 보존이 중요한 구조
  - 중첩 팀 불가 → 현재 QA-agent가 Expert 결과를 검증하는 구조가 깨짐
  - 작업 상태 지연 가능 → dependency_graph 기반 병렬 실행에 리스크

  에이전트 팀 전환이 실제로 유리해지는 조건:
    - Expert들이 서로 결과를 논의하며 방향을 맞춰야 할 때
    - 에이전트 팀이 stable 버전으로 업그레이드될 때
    - CEO 개입 없이 자율 실행이 목표일 때

  현재 ai_company의 "구조화된 파이프라인 + CEO 승인 게이트" 방식에는 Subagent가 더
   적합합니다. 에이전트 팀은 오버엔지니어링이 될 가능성이 높습니다.
